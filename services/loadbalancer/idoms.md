# Projectz Layer 4 Load Balancer: C++ Idioms and Architectural Standards

To achieve the sustained throughput requirement of 250,000 connections per second, the C++ codebase must strictly adhere to the following idioms and system engineering standards. These paradigms ensure memory safety, prevent resource leaks, and eliminate CPU context-switching bottlenecks.

## 1. Resource Acquisition Is Initialization (RAII)

Raw file descriptors generated by POSIX system calls (`socket()`, `epoll_create1()`, `accept()`) must never be managed manually. They must be encapsulated within a class boundary.

- **Standard:** The constructor acquires the file descriptor. The destructor strictly calls `close()`.
- **Rationale:** In a high-concurrency environment, connection drops and protocol errors are frequent. RAII guarantees deterministic resource cleanup during stack unwinding, preventing file descriptor exhaustion.

## 2. Move Semantics Over Copy Semantics

Network sockets represent unique kernel-level resources. They cannot be duplicated in user space.

- **Standard:** All classes managing file descriptors or network buffers must explicitly delete copy constructors and copy assignment operators (`= delete`). Ownership must be transferred between threads or connection pools exclusively using move constructors and move assignment operators (`&&`).
- **Rationale:** This enforces strict ownership rules at compile-time and prevents "double-close" bugs where two separate objects attempt to terminate the same socket.

## 3. Exception-Free Hot Paths

The C++ exception handling mechanism introduces severe latency penalties (stack unwinding) when an exception is thrown.

- **Standard:** The `try`/`catch`/`throw` paradigms are strictly prohibited within the `epoll_wait` event loop. Error handling in the hot path must be managed using `std::expected` (C++23) or standard return codes. Exceptions are only permitted during the initial configuration phase (e.g., failing to bind to the listening port).
- **Rationale:** Dropped packets and terminated sockets are expected states in Layer 4 routing, not exceptional anomalies. Deterministic error handling maintains a flat execution profile.

## 4. Deterministic Memory Layout

Heap allocation (`new`, `malloc`, `std::vector::push_back`) requires synchronization mechanisms inside the operating system's memory manager, creating a severe bottleneck across multiple threads.

- **Standard:** The `epoll_event` buffer used to capture kernel notifications must be stack-allocated using contiguous standard library containers, specifically `std::array`.
- **Rationale:** Stack allocation bypasses the OS memory manager entirely. The memory is instantly allocated when the function frame is created and perfectly aligned for the CPU's L1 cache, minimizing cache misses during event iteration.

## 5. Non-Blocking Sockets and Edge-Triggered Events (`EPOLLET`)

The interaction between the user-space application and the Linux kernel must be optimized to reduce system call volume.

- **Standard:** All sockets (listening and client) must have the `O_NONBLOCK` flag applied via `fcntl()`. The `epoll` instance must register descriptors using the `EPOLLET` (Edge-Triggered) flag.
- **Rationale:** Level-Triggered mode spams the application with notifications as long as the read buffer contains data. Edge-Triggered mode notifies the application only once when the state changes. The application must then read the socket in a loop until `EAGAIN` or `EWOULDBLOCK` is returned, signifying the buffer is drained.

## 6. C++23 Modules over Header Files

To practically eliminate compilation macro-collisions, enforce pristine abstraction barriers, and massively speed up incremental build times across the monorepo, standard header `#include` paradigms are deprecated for internal logic.

- **Standard:** All new business logic, abstractions (like `FileDescriptor`), and application states must be encapsulated inside C++23 Modules (`.cppm` files) and utilized via the `import` keyword. Legacy system headers (`<unistd.h>`, `<sys/epoll.h>`) must be strictly constrained within the "Global Module Fragment" (`module;`) at the top of the module file.
- **Rationale:** Traditional headers `#include` code directly, exposing private internal macros and forcing the compiler to re-parse the same file thousands of times. Modules are compiled exactly once into a binary interface format, guaranteeing flawless ODR (One Definition Rule) compliance and significantly decreasing CI/CD pipeline build times.
